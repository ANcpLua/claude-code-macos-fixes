Why Computer Use says not yet granted — and the fix

TL;DR — The built-in computer-use MCP server lives inside the claude CLI process. macOS TCC attributes Accessibility and Screen Recording to the binary that asks — claude.exe (yes, really) — not to your terminal and not to Claude.app. Drag that binary into both permission lists, then restart the CLI (claude --resume keeps your session): grants are only read at process start.

Why every toggle you flipped changed nothing

You enabled Ghostty, iTerm, Terminal, even Claude.app — all blue toggles — and request_access still loops with “Accessibility and Screen Recording permission(s) not yet granted.” Two macOS facts explain it:

Fact 1 · TCC binds to the requesting binary not the host app

The computer-use server is not a separate helper app. It runs inside the CLI process, so the TCC identity macOS checks is the CLI binary itself:

/opt/homebrew/lib/node_modules/@anthropic-ai/claude-code/bin/claude.exe

Terminal-app grants cover other things — your shell’s screencapture may even work while the MCP server stays blocked, which makes the failure genuinely confusing.

Fact 2 · Grants are read once, at launch stale-process trap

A running process caches its TCC state. Fixing the lists mid-session changes nothing until the claude process restarts. This is why “I granted it and it still fails” is the default experience.

The fix, step by step

1 · Locate your real binary Terminal 
readlink -f $(which claude)
# → …/@anthropic-ai/claude-code/bin/claude.exe

Homebrew shown; npm/bun installs land elsewhere — this command always tells the truth.

2 · Reveal it in Finder open -R 
open -R "$(readlink -f $(which claude))"

Spotlight does not index /opt/homebrew, so the search field in the permission dialog will never find it. Finder is the reliable route.

3 · Drag it into BOTH permission lists System Settings

System Settings → Privacy & Security → Accessibility: drag claude.exe from Finder straight onto the app list. Repeat for Screen & System Audio Recording. Enable both toggles. Drag & drop skips the broken “+” dialog entirely.

4 · Restart the CLI — context survives claude --resume 
Ctrl+C
claude --resume   # pick your session — nothing is lost
5 · Request access again in the session

Have Claude call request_access once more. Instead of the loop message you get the one-time per-app approval dialog. Approve it — computer use is live.

The five traps (we hit the first four ourselves)

Trap 1 · The Spotlight search field “+” file dialog

In the “+” dialog, Cmd+Shift+G only opens Go-to-Folder after you click into the file list; otherwise your pasted path lands in the search box — and Spotlight doesn’t index /opt/homebrew, so it finds nothing. Drag & drop from Finder avoids the dialog completely.

Trap 2 · The Unarchiver eats your CLI double-click

Don’t double-click claude.exe. macOS treats .exe as a Windows archive and hands it to The Unarchiver, which offers to “extract” your CLI. Cancel — the file is dragged, never opened. (The binary is named .exe on macOS too; build-system artifact.)

Trap 3 · Granting the wrong app Ghostty / iTerm / Claude.app

Terminal and Claude.app entries cover other features. The cruellest variant: shell screencapture works through the terminal’s grant while the MCP server stays blocked — so quick sanity checks “prove” permissions are fine when they aren’t.

Trap 4 · The stale process grants read at launch

Everything correct, still failing? The session predates the grant. Ctrl+C then claude --resume — TCC state is only read at process start, and resume keeps your full context.

Trap 5 · The orphaned per-version helper community-reported · unverified by us

All TCC grants correct, restart done, still looping? Several reports (anthropics/claude-code#50735 plus duplicates #48311, #49282, #41606) point at Claude Code’s per-version helper bundle with a .verified sentinel under ~/Library/Application Support/Claude/claude-code/<VERSION>/ not being re-provisioned by auto-updates. Their reliable fix:

ls ~/Library/Application\ Support/Claude/claude-code/
# last resort: quit CLI, delete the claude-code/
# dir, relaunch — it re-extracts

Caveat from our own machine: a helper dir three versions behind the CLI (2.1.209 vs 2.1.212) worked fine here once Traps 1–4 were fixed — a version mismatch alone proves nothing. Rule out Traps 1–4 first; deletion is a last resort, not a diagnostic.

FAQ

Why is it called claude.exe on a Mac?
Build-system artifact — it’s the genuine native macOS binary, the name just kept the .exe suffix. Ignore it (and don’t let The Unarchiver near it).
Do I lose my session when I restart?
No. claude --resume restores the full conversation context; you continue exactly where you stopped.
Does this apply to the Claude desktop app?
No — Claude.app manages its own TCC entries. This guide is for the CLI (claude in a terminal), where the built-in computer-use MCP inherits the CLI binary’s identity.
I use npm/bun, not Homebrew — same fix?
Yes, only the path differs. readlink -f $(which claude) in step 1 resolves your actual install location; drag whatever it prints.
Claude Code updated — do I need to redo this?
If the update replaces the binary at the same path, existing grants usually survive. If computer use starts looping again after an update, repeat steps 1–4 — two minutes.
